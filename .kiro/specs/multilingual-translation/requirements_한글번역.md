# 요구사항 문서 (한글 해설판)

## 소개

BADA 다국어 번역 기능은 외국인 노동자가 분석 결과(타임라인, 공제 내역, 미지급 의심 금액, 면책 고지)를
자신의 모국어로 이해할 수 있도록 동적 콘텐츠를 번역하는 시스템이다.
Amazon Translate를 기본 엔진으로 사용하며, 크메르어·네팔어 같은 저자원 언어에는 영어 피벗 + Claude 보정을 적용한다.
원문은 항상 병기하여 번역 오류 시 확인이 가능하도록 한다.

## 용어 설명

| 용어 | 뜻 |
|------|-----|
| Translator | Amazon Translate API를 감싸서 텍스트 번역을 수행하는 모듈 |
| TranslationService | 분석 결과에서 번역할 대상을 뽑아내고, 원문-번역 대조표를 만드는 서비스 |
| TimelineService | 타임라인 이벤트 설명을 모국어로 번역해서 저장하는 서비스 |
| LanguageConfig | 지원 언어 목록, 언어 코드 매핑, 번역 전략을 한곳에서 관리하는 설정 모듈 |
| TranslationPair | 원문(source_text)과 번역문(translated_text)을 함께 보관하는 데이터 구조 |
| LanguageStrategy | 각 언어에 대해 "어떤 방식으로 번역할지"를 정의하는 데이터 클래스 |
| Pivot Translation | 한국어→영어→대상언어로 2단계 거쳐 번역하는 경로 |
| Graceful Degradation | API가 실패해도 시스템이 죽지 않고, 원문을 대신 반환하는 전략 |
| SUPPORTED_LANGUAGES | 시스템이 지원하는 언어 8개: ko, vi, en, id, km, ne, th, ja |

---

## 요구사항 목록

### 요구사항 1: 단일 텍스트 번역

**사용자 스토리:** 외국인 노동자로서, 한국어 분석 결과를 모국어로 번역받고 싶다. 법률 상담 전에 내 사건 내용을 이해하기 위해서.

**이게 완성되려면 (합격 기준):**

1. 비어있지 않은 한국어 텍스트 + 지원 언어가 들어오면 → 해당 언어로 번역된 문자열을 반환해야 한다
2. 대상 언어가 "ko"(한국어)면 → 원문 그대로 반환 (번역할 필요 없으니까)
3. 빈 문자열이 들어오면 → 빈 문자열 반환, API 호출 안 함
4. 지원하지 않는 언어 코드가 들어오면 → UnsupportedLanguageError 에러 발생
5. 번역 과정에서 입력 원문 객체를 절대 수정하면 안 됨

---

### 요구사항 2: 배치 번역 (여러 개 한꺼번에)

**사용자 스토리:** 시스템 운영자로서, 여러 텍스트를 한번에 번역하고 싶다. 네트워크 호출 횟수를 줄이고 성능을 높이기 위해서.

**합격 기준:**

1. 텍스트 리스트 + 대상 언어가 들어오면 → 같은 길이의 번역 리스트를 반환
2. 순서 보존: i번째 입력 → i번째 출력 (뒤섞이면 안 됨)
3. 대상 언어가 "ko"면 → 입력 리스트 그대로 반환, API 호출 없음
4. 배치 중 하나가 실패해도 → 그 항목만 원문 반환, 나머지는 계속 번역

---

### 요구사항 3: 언어별 번역 전략 자동 적용

**사용자 스토리:** 시스템 설계자로서, 언어마다 최적 번역 전략이 자동 적용되길 원한다. 저자원 언어도 품질을 확보하기 위해서.

**합격 기준:**

1. 베트남어(vi), 영어(en), 인도네시아어(id), 태국어(th), 일본어(ja) → "직접" 전략 (한국어→대상어 바로 번역)
2. 크메르어(km), 네팔어(ne) → "피벗+보정" 전략 (한국어→영어→대상어 + Claude가 다듬기)
3. 피벗 번역 시 → 먼저 한국어→영어, 그 다음 영어→대상어 순서
4. Claude 보정 적용 시 → 원본 한국어 + 기계번역 초안을 Claude에게 보내서 자연스럽게 고침
5. Claude 보정이 실패하면 → 보정 안 한 기계번역 초안을 그대로 사용 (안 죽음)

---

### 요구사항 4: 그레이스풀 디그레이데이션 (우아한 실패 처리)

**사용자 스토리:** 외국인 노동자로서, 번역 API가 실패해도 결과를 볼 수 있길 원한다. 시스템 장애 때도 원문으로라도 확인하기 위해서.

**합격 기준:**

1. Amazon Translate API가 어떤 에러로든 실패하면 → 원문 그대로 반환 + 경고 로그
2. 번역 타임아웃 발생 시 → 원문 반환, 예외를 호출자에게 던지지 않음
3. 실패 시 TranslationPair에는 → source_text와 translated_text 모두 원문이 들어감
4. 번역 실패로 인한 예외가 파이프라인까지 전파되면 절대 안 됨

---

### 요구사항 5: 원문-번역 대조표 생성

**사용자 스토리:** 외국인 노동자로서, 공제 내역과 미지급 의심 금액을 원문과 함께 모국어로 확인하고 싶다. 상담기관 방문 시 정확한 내용을 전달하기 위해서.

**합격 기준:**

1. 분석 결과에 공제 항목이 있으면 → 각 항목마다 번역 쌍(TranslationPair)을 만든다
2. 미지급 의심 금액이 0보다 크면 → 미지급 설명 번역 쌍을 만든다
3. **면책 고지는 항상 포함** — 다른 내용이 없어도 반드시 들어간다
4. 모든 번역 쌍의 source_text는 비어있으면 안 된다
5. 모든 번역 쌍의 translated_text도 비어있으면 안 된다 (실패 시 원문으로 채움)
6. 각 쌍에 evidence_type(증거유형), related_issue(관련이슈) 메타데이터를 붙인다

---

### 요구사항 6: 타임라인 이벤트 번역

**사용자 스토리:** 외국인 노동자로서, 사건 타임라인의 각 이벤트 설명을 모국어로 읽고 싶다. 사건 경과를 시간순으로 이해하기 위해서.

**합격 기준:**

1. 타임라인 이벤트에 description(설명)이 있으면 → 대상 언어로 번역해서 description_translated에 저장
2. 대상 언어가 "ko"면 → description_translated = description (같은 값)
3. 번역을 해도 → 원본 한국어 description 필드는 절대 수정 안 함
4. 번역 실패 시 → description_translated에 원본 한국어 description을 넣음

---

### 요구사항 7: 지원 언어 관리

**사용자 스토리:** 시스템 관리자로서, 지원 언어와 번역 전략을 한곳에서 관리하고 싶다. 새 언어 추가 시 일관된 방식으로 확장하기 위해서.

**합격 기준:**

1. 정확히 8개 언어를 지원한다: ko, vi, en, id, km, ne, th, ja
2. 언어 코드를 조회하면 → 해당 언어의 전략 정보를 반환 (코드, 번역 방식, 피벗 여부 등)
3. 지원하지 않는 코드를 조회하면 → UnsupportedLanguageError 발생
4. vi, en, id, th, ja에는 "direct" 전략을 배정
5. km, ne에는 "pivot+refine" 전략을 배정 (피벗 언어 = 영어)
6. ko에는 "none" 전략을 배정 (번역 안 함)

---

### 요구사항 8: 원문 병기 원칙

**사용자 스토리:** 외국인 노동자로서, 번역 결과 옆에 항상 한국어 원문이 함께 표시되길 원한다. 번역이 부정확할 때 상담기관에서 원문을 참조하기 위해서.

**합격 기준:**

1. TranslationPair에는 항상 source_text(한국어 원문) + translated_text(번역문) 둘 다 있어야 함
2. TimelineService도 항상 description(한국어) + description_translated(번역) 둘 다 제공
3. 번역 콘텐츠를 보여줄 때 → 원문과 번역을 나란히 표시 (side-by-side)
4. 어떤 번역 과정에서도 원본 한국어 텍스트를 버리거나 덮어쓰면 절대 안 됨

---

### 요구사항 9: 초장문 텍스트 처리

**사용자 스토리:** 시스템 운영자로서, Amazon Translate API 제한(10KB)을 초과하는 긴 텍스트도 정상 번역되길 원한다. 어떤 길이의 분석 결과도 누락 없이 번역하기 위해서.

**합격 기준:**

1. 텍스트가 10,000바이트를 초과하면 → 문장 경계에서 나눈다
2. 나눈 각 조각을 개별 번역한 후 → 순서대로 이어붙여서 반환
3. 조각 하나가 번역 실패하면 → 그 조각만 원문 사용, 나머지는 계속 번역

---

### 요구사항 10: 리포트 실시간 다국어 조회 (추가됨)

**사용자 스토리:** 외국인 노동자로서, 분석을 다시 실행하지 않고도 리포트를 모국어로 볼 수 있길 원한다. 언어를 바꿔가며 내용을 확인하기 위해서.

**합격 기준:**

1. 리포트 조회 시 lang 파라미터가 ko가 아니면 → 타임라인, 공제, 대조표, 누락 안내를 그 자리에서 Amazon Translate로 번역해서 보여줌
2. lang=ko이면 → 모든 내용을 한국어 원문 그대로 표시, 번역 API 호출 안 함
3. 리포트 UI 요소(섹션 제목, 라벨, 면책 고지)도 lang에 따라 미리 정의된 텍스트로 전환
4. **분석을 다시 돌리지 않아도** 다른 언어로 리포트를 볼 수 있어야 함
5. 분석 결과는 한국어로만 DB에 저장, 번역은 조회 시점에 수행

---

### 요구사항 11: 제출용 PDF는 한국어 고정 (추가됨)

**사용자 스토리:** 외국인 노동자로서, 인쇄/PDF 저장 시 한국어 원본 리포트를 받고 싶다. 공공기관(고용노동부)이나 법무법인에 제출하기 위해서.

**합격 기준:**

1. 리포트에서 인쇄/PDF 버튼을 누르면 → 한국어(lang=ko) 버전의 리포트가 열림
2. 인쇄 버튼은 현재 어떤 언어로 보고 있든 **항상 lang=ko** 리포트로 연결
3. 한국어 리포트에는 동적 콘텐츠에 번역이 적용되지 않고, 원문 그대로 표시

---

### 요구사항 12: 번역 모드 독립 설정 (추가됨)

**사용자 스토리:** 개발자로서, 번역 기능만 AWS 모드로 전환할 수 있길 원한다. LLM/OCR이 미구현이어도 번역 테스트를 하기 위해서.

**합격 기준:**

1. `TRANSLATE_MODE` 환경변수를 지원하며, 번역에 한해서만 PROVIDER_MODE를 오버라이드함
2. `TRANSLATE_MODE=aws` 설정 시 → PROVIDER_MODE 값과 무관하게 AmazonTranslator 사용
3. `TRANSLATE_MODE`를 설정하지 않으면 → PROVIDER_MODE 값을 따름
