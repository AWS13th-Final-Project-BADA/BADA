## 슬라이드 4 — 핵심 설계 원칙 (차별화 포인트) (2:30–4:00)
[화면]
- 계산·비교·정렬·판정 = 규칙 기반 코드 (worker/rules, 11개 모듈)
- 문장화·요약·OCR = LLM (worker/llm)
- "AI가 금액을 잘못 계산하면?" → "계산은 AI가 안 합니다."
- 뒷받침: property-based test 13개 + 가드레일(환각 숫자 차단)

[멘트]
"이 프로젝트에서 가장 중요한 설계 결정입니다. 금액 계산, 비교, 정렬, 판정 같은 건 전부 규칙 기반 코드가 합니다. LLM은 OCR로 글자를 읽고, 사실을 문장으로 정리하는 일만 합니다. 그래서 'AI가 금액을 잘못 계산하면 어떡하냐'는 질문에, 저희는 '계산은 AI가 하지 않는다'고 답할 수 있습니다. 규칙 엔진은 property-based 테스트 13개로 불변식을 검증하고, 요약이 근거에 없는 숫자를 만들어내면 가드레일이 사실 기반으로 되돌립니다. 예를 들어 입금 내역이 없으면 '0원 수령'으로 단정하지 않고 '확인 필요'로 남깁니다."

## 슬라이드 7 — 데모 2: 증거 업로드 & 추출 (6:00–8:00)
[라이브 / fallback 스크린샷]
- 이미지(명세서/계약서/통장/카톡) + 오디오 업로드
- OCR(Bedrock Vision) / STT(Transcribe) → 엔티티 추출 → 사용자 검토·수정(HITL)

[멘트]
"증거를 올립니다. 이미지는 Bedrock Vision으로 OCR을 돌려 시급, 근무시간, 공제 같은 항목을 구조화하고, 녹취 파일은 Transcribe로 전사한 뒤 같은 방식으로 엔티티를 뽑습니다. 여기서 핵심은, 자동 추출 결과를 사용자가 직접 확인하고 고칠 수 있다는 점입니다. 사람이 최종 확인하는(HITL) 구조라서, OCR이 애매하게 읽은 값을 그대로 분석에 넣지 않습니다."

## 슬라이드 9 — 아키텍처 (10:00–12:00)
[화면 — 다이어그램 1장]
- Mobile → WAF → ALB(HTTPS·ACM·TLS1.3) → Backend(FastAPI, ECS Fargate)
- Backend → RDS(PostgreSQL+PostGIS), S3(SSE-KMS), SQS
- SQS → Worker(Fargate Spot) → Bedrock / Transcribe / Translate
- 3-tier 격리: ALB=public / ECS=private / RDS=private (+ 단일 NAT egress, 토글로 롤백·종료)

[멘트]
"전체 아키텍처입니다. 모바일에서 온 요청은 WAF와 ALB를 거쳐 FastAPI 백엔드로 들어오고, 백엔드는 데이터를 RDS와 S3에 저장한 뒤 무거운 분석은 SQS 큐에 넘깁니다. 워커가 큐를 받아 OCR·전사·번역·분석·PDF를 비동기로 처리합니다. 전 구간을 ALB는 퍼블릭, ECS와 RDS는 프라이빗 서브넷으로 3계층 격리했고, 프라이빗 아웃바운드는 NAT로 처리합니다. 이 전환은 토글로 되어 있어서 문제가 생기면 즉시 되돌리고, 프로젝트 종료 때는 꺼서 과금을 멈춥니다."

## 슬라이드 10 — 비동기 파이프라인 & 신뢰성 (12:00–14:00)
[화면]
- 접수→S3→SQS→Worker 비동기 분리(사용자는 기다리지 않음)

[멘트]

## 슬라이드 11 — 엔지니어링 성숙도 (14:00–15:30)
[화면 — 숫자 위주]
- 테스트 ~218개(backend ~50 + worker ~168) + property-based 13개
- CI: ruff + bandit(SAST) + pip-audit(SCA) + pytest-cov + eval 하네스 회귀
- CD: GitHub Actions OIDC → ECR → ECS 롤링 + 서킷브레이커 롤백, `infra/**`는 PR에서 read-only plan 코멘트
- Auto Scaling 실증(k6): Backend CPU 1→2, Worker SQS 적체 1→3 scale-out
- 성능 개선: OCR 병렬 12건 240s→25s, PDF 50s→15s

[멘트]
"엔지니어링 성숙도입니다. 테스트는 약 218개, 그중 규칙 엔진에는 property-based 테스트 13개가 붙어 있습니다. CI는 린트, 보안 스캔, 의존성 취약점 검사, 커버리지, 그리고 정확도 회귀까지 PR마다 돌립니다. 배포는 OIDC로 키 없이 하고, 인프라 변경은 PR에 read-only plan을 자동으로 달아 리뷰합니다. 오토스케일링은 k6 부하 테스트로 실제로 발동시켜서, 백엔드는 CPU 기준으로, 워커는 SQS 적체 기준으로 태스크가 늘어나는 걸 그래프로 확인했습니다. 성능도 눈에 띄게 개선했는데, 예를 들어 12건 OCR을 병렬화해서 240초를 25초로 줄였습니다."

[발표 팁] 여기서 Grafana scale-out 그래프 캡처를 띄우면 임팩트가 큼.

## 부록 A — 예상 질문 & 답변
- Q. AI가 금액을 틀리면? → 계산은 규칙 엔진이 전담, LLM은 문장화만. property-based test 13개 + 가드레일. 입금내역 없으면 0원으로 단정 안 하고 '확인 필요'.
- Q. 개인정보·국외이전은? → 처리방침·동의·국외이전 요건을 `docs/decisions/privacy-legal-requirements.md`로 정리. 데모/기간상 구현은 보류, 실서비스 전 최우선으로 명시.
- Q. 회원 탈퇴/데이터 삭제는? → 현재 사건·증거 단위 삭제는 있으나 계정 완전 삭제(S3 원본 포함)와 자동 파기는 실서비스 전 과제로 남김(정직하게).
- Q. 비용/확장성은? → 예산 $55/$1,500. Worker는 Fargate Spot, 오토스케일링은 CPU·SQS 적체 기준. k6로 scale-out 실증.
- Q. 왜 K8s/Kafka 안 썼나? → 팀 규모·기간·운영 부담 대비 과함. ECS Fargate + SQS로 충분. (금지 스택 사유는 steering 문서에 기록)
- Q. Cognito는? → 실제 인증은 자체 소셜 OAuth + JWT. Cognito 리소스는 레거시/병행이며 활성 경로 아님.

[주의] 모델 벤치마크(`docs/model-benchmark.md`)를 언급할 경우: 10케이스 스키마 통과율이 모델별로 갈리고(예: Opus가 가장 높음) 프로덕션 모델과의 차이를 설명할 수 있어야 함. 준비 안 됐으면 이 슬라이드/주제는 꺼내지 않는 편이 안전.
